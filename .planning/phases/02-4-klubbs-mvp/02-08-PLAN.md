<objective>
**Deferred Phase 2.1 sub-plan — NOT for 2026-05-20 4-klubbs.**

Minimal auth so sekretariat-helpers can pitch in from their phones at the
event without exposing the bridge to the open LAN. The operator generates
a short, memorable code (`sänkan42`, `stenen17`, `branten99`) drawn from
the canonical SOFT 2024 kontrollbeskrivningar wordlist, shouts it across
the parking lot, and helpers type it on `/access` on their phones.
Anyone with the code can register runners; nobody else can.

## Trust model

- **Closed LAN deployment.** The bridge sits on the same Wi-Fi the club
  is using for the event. Not exposed to the public internet.
- **Per-event rotation.** Codes are scoped to a single competition_id
  and expire ~24h after `competition.date`. A code leaked in May does
  nothing in June.
- **Limited blast radius.** A valid code only grants the ability to ADD
  registrations (POST /api/competitors, walkup, queue-advance). It does
  NOT grant: deletes, edits, exports, admin endpoints, settings, MIP/MOP
  routes, the Tweaks panel, or any read endpoint that returns PII.
- **No accounts.** All helpers look identical in the audit trail (the
  event_codes table records which code was used, not who used it). For
  small training events this is fine; if you need per-person audit
  trails you're past Phase 2.1 scope and should be building real users.

## Code format + entropy

- **Format:** `<word>-<NNN>` where `word` ∈ EVENT_CODE_WORDS (35 entries)
  and `NNN` ∈ [100..999] (900 values, NEVER zero-padded). Examples:
  `sänkan-127`, `branten-403`, `röset-999`, `bron-555`, `gropen-100`.
- **Why the floor at 100:** every spoken number is exactly 3 syllables
  ("fyra-fem-sex" or "fyrahundrafemtiosex"). The user gave the operative
  example: `dungen-noll-noll-sju` (007 zero-padded) dies at the start
  line — too many `noll`s in a row gets misheard as a single null and
  the helper types `dungen-7`, which never matches.
- **Why a dash:** clear separator in spoken form ("dungen STRECK
  fyra-fem-sex") and in the input box (avoids ambiguous parses like
  `sänkan42` where the helper might type `sänkan 42` with a space).
- **Entropy:** 35 × 900 = 31 500 combinations ≈ **14.9 bits**. Still
  weak in the absolute sense but the +3 bits over the original
  `<word><NN>` design buys real margin against the rate-limit math
  below.
- **Defense-in-depth:** per-IP rate limit (10 attempts / 60 s + exp.
  backoff after 5 fails). At sustained 10 attempts/min, exhausting
  31 500 codes takes ~52 h — well past the 24 h expiry. Even if the
  rate-limit fails entirely and an attacker pushes 100 attempts/min,
  exhaustion is ~5.2 h, leaving < 1/5 of the event-day window for an
  attacker to land — still survivable for the trust model above.
- **If you ever expose the bridge over the public internet,** 15 bits
  is NOT enough, even with the rate limit. The ADR explicitly calls
  this out and gates expansion on either real auth or a longer code.

## Why orienteering wordlist (not random PIN)

- Memorable when shouted across a parking lot (`sänkan-127` vs `8473-91`)
- Domain-flavored — feels like part of the sport, not a corporate auth
  product
- Curated to avoid awkward pronounceability (no compounds like
  `stenmurskorsningen`, no homophones, no 3+ syllable words)
- Locked list of 35 words (see EVENT_CODE_WORDS in must_haves) drawn
  from SOFT 2024 kontrollbeskrivningar across all five symbol categories
  (Topografi / Sten / Vatten / Vegetation / Människo) — source PDF at
  `.reference/d5561844-Kontrollbeskrivningar_2024__20240112.pdf`

## Out of scope (push to Phase 3+)

- Per-helper named accounts + per-action audit trail
- QR-code rendering of the access URL+code
- SMS/email distribution of codes
- Multi-factor auth on the admin-side generator
- Encryption at rest for the signing secret (matches ADR-0008 trust
  model — single-user laptop, full-disk encryption recommended)
  </objective>
