<objective>
**Deferred Phase 2.1 sub-plan — NOT for 2026-05-20 4-klubbs.**

Minimal auth so sekretariat-helpers can pitch in from their phones at the
event without exposing the bridge to the open LAN. The operator generates
a short, memorable code (`sänkan42`, `stenen17`, `branten99`) drawn from
the canonical SOFT 2024 kontrollbeskrivningar wordlist, shouts it across
the parking lot, and helpers type it on `/access` on their phones.
Anyone with the code can register runners; nobody else can.

## Trust model

- **Closed LAN deployment.** The bridge sits on the same Wi-Fi the club
  is using for the event. Not exposed to the public internet.
- **Per-event rotation.** Codes are scoped to a single competition_id
  and expire ~24h after `competition.date`. A code leaked in May does
  nothing in June.
- **Limited blast radius.** A valid code only grants the ability to ADD
  registrations (POST /api/competitors, walkup, queue-advance). It does
  NOT grant: deletes, edits, exports, admin endpoints, settings, MIP/MOP
  routes, the Tweaks panel, or any read endpoint that returns PII.
- **No accounts.** All helpers look identical in the audit trail (the
  event_codes table records which code was used, not who used it). For
  small training events this is fine; if you need per-person audit
  trails you're past Phase 2.1 scope and should be building real users.

## Entropy + brute-force

- Wordlist: ≥40 curated words. Number: 0-99 (100 values). Total: ≥4 000
  combinations ≈ **12 bits**. Weak in the absolute sense.
- Defense-in-depth: per-IP rate limit (10 attempts / 60s + exponential
  backoff after 5 fails). At 10 attempts/min, exhausting 4 000 codes
  takes ~6.7h — well past the event window. Combined with the 24h
  expiry this is fit-for-purpose for the trust model above.
- If you ever expose the bridge over the public internet, the rate
  limit plus 12 bits is NOT enough. The ADR explicitly calls this out.

## Why orienteering wordlist (not random PIN)

- Memorable when shouted across a parking lot (`sänkan42` vs `847291`)
- Domain-flavored — feels like part of the sport, not a corporate auth
  product
- Curated to avoid awkward pronounceability (no compounds like
  "stenmurskorsningen", no homophones)
- Source: SOFT 2024 kontrollbeskrivningar (the canonical Swedish
  control-description reference) — see `.reference/` for the PDF

## Out of scope (push to Phase 3+)

- Per-helper named accounts + per-action audit trail
- QR-code rendering of the access URL+code
- SMS/email distribution of codes
- Multi-factor auth on the admin-side generator
- Encryption at rest for the signing secret (matches ADR-0008 trust
  model — single-user laptop, full-disk encryption recommended)
  </objective>

<verification>

Goal-backward checks:

- **A helper with a phone can register a runner in under 30 seconds from
  scratch:** operator generates code → SMSes to helper → helper opens
  fartol-ip:3000/access on phone → types code → lands on /registration
  → adds a runner. End-to-end time < 30s.
- **An attacker on the same LAN WITHOUT a code cannot add registrations:**
  the preHandler gate returns 401 on every write attempt without a
  valid cookie.
- **Brute force is infeasible within an event window:** the rate-limit
  - 12-bit entropy + 24h expiry math is documented in ADR-0010 and
    validated by the manual probe (Task 6 step 5).
- **Operator on the desk laptop is never blocked:** localhost bypass +
  the existing FARTOL_DEV admin-route conventions.

</verification>
