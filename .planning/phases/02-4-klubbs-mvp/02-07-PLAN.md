<objective>
**Deferred sub-plan — NOT for the 2026-05-20 production gate.**

UI-driven secrets management. Today every API key (Eventor, soon
Livelox and Liveresultat) has to be set via `~/.env.fartol` and
`source`d into the shell. That works for Linux operators. It does NOT
work for Windows operators (the target user base for cross-club
distribution), who don't touch dotfiles.

This plan adds an "Inställningar" (Settings) page that reads and writes
the keys from the existing `config` SQLite table. Boot precedence is
preserved: `process.env.X` wins so CI / headless / advanced operators
keep working unchanged.

Drivers:

1. **Windows operator UX** — clicking through a UI is the only viable
   path for non-CLI operators (4-klubbs runs Linux today; phase 2.1
   distribution to other clubs needs Windows support).
2. **Multi-key future** — Livelox + Liveresultat keys arrive in Phase 3.
   Per-integration shell exports don't scale; a single "Inställningar"
   surface that lists every known integration does.
3. **Operator self-service** — rotating a key today requires editing
   `~/.env.fartol` + restarting the bridge. Through the UI, save +
   refresh, no shell.

Out of scope (deferred to later phases):

- Encryption at rest — plain text in `config` table matches our existing
  trust model (single-user laptop, full-disk encryption recommended per
  ADR-0008). Re-evaluate if/when multi-tenant or cloud-hosted.
- OAuth-style flows for integrations that support them — Eventor / Livelox
  / Liveresultat all use static API keys, so OAuth is YAGNI for now.
- Per-competition key scoping — single global key per integration.
  </objective>
