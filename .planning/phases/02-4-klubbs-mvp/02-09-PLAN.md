## Problem

Phase 02-01 declared a partial UNIQUE index on `eventor_competitors.si_card`
based on the assumption that each SI card belongs to exactly one runner in
the Eventor federation database. That assumption does not survive contact
with real data — when an operator wires a valid EVENTOR_API_KEY and the
boot scheduler downloads the cachedcompetitors archive, ingest crashes
with:

```
SqliteError: UNIQUE constraint failed: eventor_competitors.si_card
  at ingestEventorCache (apps/edge/src/eventor/cache.ts)
```

In production data, SI cards are duplicated for several legitimate reasons:

1. **Family-shared cards** — a parent registers a child on the same card.
2. **Card reassignment** — a runner lost their card, ordered a
   replacement, and the federation's record-keeping kept BOTH personIds
   pointing at the original card number.
3. **Rental cards** — clubs return rental cards to a shared pool and
   different runners check out the same physical card across seasons;
   each rental gets logged against a different personId.
4. **Data quality** — the federation database is decades old; manual
   entry errors persist.

These duplicates are real-world properties of Eventor data, not corruption.
A correct system must tolerate them. The IOF spec (Eventor REST `cachedcompetitors`)
makes NO uniqueness promise on `ControlCard`; the assumption was ours alone.

## Why this is the 4-klubbs MVP blocker

Eventor cache feeds the walk-up autocomplete (REQ-EVT-CMP-005 — auto-assign
a punched card to its registered runner). On 2026-05-20 at the 4-klubbs
training, an operator inserts an SI card at the registration desk; FartOL
looks up `si_card` in the cached federation data and pre-fills the runner's
name, club, and birth year so the operator just types nothing and clicks
Spara. Without the cache, the desk falls back to typing the runner's name
from scratch — which the runbook explicitly calls out as the slow path
that bottlenecks at the 60-runner mark.

Right now the cache is empty because every ingest attempt aborts on the
first duplicate. The desk has no autocomplete. Phase 2.0's walk-up flow
cannot work end-to-end until this is fixed.

## Decision: tri-state lookup with context-aware disambiguation

Drop the UNIQUE constraint; let `eventor_competitors` accept multiple rows
per si_card. Move the disambiguation logic into the `lookupBySiCard`
function, which now takes an optional `activeCompetitionId` parameter
(the walk-up modal always has one; the diagnostic /api/eventor/lookup
admin endpoint may not).

Selection rule, in order:

1. **Context match wins.** If `activeCompetitionId` is supplied AND one
   of the Eventor candidates is already registered in the local
   `competitors` table for that competition (matched via `si_card`),
   return that one. This is the dominant path: most of the time the
   runner walking up has pre-registered through the entries import.

2. **Recency wins.** If no context match exists, return the candidate
   with the largest `modify_date_ms`. The Eventor federation updates
   the modifyTime when a person changes club, renews their license, or
   re-binds a card — the freshest record is the most likely
   currently-correct one.

3. **Surface ambiguity.** Always return `alternatives: N - 1` (count of
   candidates not picked) so the UI can render a "+N andra" chip; tapping
   it lets the operator override the pick without re-typing.

The 14-line `lookupBySiCard` becomes ~50 lines including the join and the
return shape, but every call site continues to work because the existing
`kind: "hit"` branch covers the unique-card case (which is the majority).

## Non-goals

- **Cleaning up the federation.** We don't fix Eventor's data. The
  duplicates are theirs to resolve (or not). FartOL tolerates them.
- **Per-event si_card uniqueness.** The local `competitors` table CAN
  enforce one card per competitor per event — that's a real constraint
  (one runner cannot start a single race on two cards) and stays as-is.
  This plan only relaxes the FEDERATION cache constraint.
- **Cross-card analytics.** Counting how often duplicates occur is a
  diagnostic-only feature; not in scope for this plan. If the next-phase
  Settings work wants a "Dubbletter på SI-bricka: N" line, it can query
  `SELECT si_card, COUNT(*) FROM eventor_competitors GROUP BY si_card HAVING COUNT(*) > 1`
  at render time without a schema change.

### Task 2 — `lookupBySiCard` tri-state with context-aware disambiguation

**Why:** The walk-up flow needs to pick the right runner when the
federation has duplicates. The function signature widens but every
existing caller continues to work via the `kind: "hit"` branch.

**Deliverables:**

1. Update `apps/edge/src/eventor/lookup.ts`:
   - New signature: `lookupBySiCard(handle: DbHandle, siCard: number, opts?: { activeCompetitionId?: string }): LookupResult`
   - New return type:
     ```ts
     type LookupResult =
       | { kind: 'hit'; competitor: EventorCompetitor; alternatives: number }
       | { kind: 'multi'; candidates: EventorCompetitor[]; reason: 'ambiguous-no-context' }
       | { kind: 'miss' };
     ```
   - Algorithm (single SQL query preferred):
     1. SELECT all `eventor_competitors` rows WHERE `si_card = ?`, LEFT JOIN
        `eventor_clubs` for the resolved `club_name`.
     2. If 0 rows → `{ kind: 'miss' }`.
     3. If 1 row → `{ kind: 'hit', competitor, alternatives: 0 }`.
     4. If >1 row AND `activeCompetitionId` supplied:
        - LEFT JOIN with `competitors` filtered by `competition_id = ?` and
          `si_card = ?`. If exactly one Eventor candidate has a matching
          local registration → that's the winner, alternatives = N-1.
        - Otherwise → recency rule: ORDER BY `modify_date_ms DESC LIMIT 1`,
          alternatives = N-1.
     5. If >1 row AND no `activeCompetitionId`:
        - Same recency rule. (We could return `kind: 'multi'` for the
          admin endpoint, but the simpler contract is "always pick a
          winner; let the caller decide whether to show alternatives".)
        - `kind: 'multi'` is reserved for a future explicit-disambiguation
          path; emit it ONLY when the algorithm cannot decide (e.g. tied
          modify_date_ms). This keeps `kind: 'multi'` as a defensive
          escape hatch, not a common code path.

2. Update `apps/edge/src/eventor/lookup.test.ts`:
   - Test: unique card → hit + alternatives:0.
   - Test: duplicate cards, no context → recency winner + alternatives:N-1.
   - Test: duplicate cards + active competition matching one → that one
     wins.
   - Test: duplicate cards + active competition matching none → recency
     winner.
   - Test: duplicate cards + tied modify_date_ms → falls through to
     `kind: 'multi'` (corner case, smaller scope than building a full
     UI for it).
   - Test: no match → miss.

3. Add fixture `apps/edge/src/eventor/__fixtures__/competitors-shared-card.xml`
   with the 3-competitor / 2-share-one-card shape described in `must_haves`.

**TDD entry:** Write the 6 test cases first against the current 14-line
lookup (they will all fail at the type level since the return shape is
different). Then update the implementation.

**Acceptance:** All 6 test cases pass; existing callers compile after
narrowing on `kind: 'hit'`.
