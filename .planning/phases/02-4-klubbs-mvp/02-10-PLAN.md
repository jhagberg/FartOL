<objective>

## Problem (UX polish, not correctness)

Today every Eventor-dependent surface asks the operator for the Eventor
event ID per click:

- `ImportRunnersView` shows a date picker → event list → import button.
  Even after a successful import, returning to the page re-renders the
  whole picker; the operator has to remember which event they used.
- The wizard's step 1 can prefill name + date from a courseData XML
  drop, but there's no Eventor-driven prefill path for events that
  exist in Eventor BEFORE courseData is finalised.
- The Tävling menu has no visual cue indicating which local
  competitions correspond to Eventor events vs. internal-only trainings.

For competitions that ARE in Eventor (sanctioned events, district
trainings registered there), this fragmentation costs ~15 seconds of
operator time per import and creates re-entry risk when the operator
returns mid-event ("did I import from event 89234 or 89235?").

## Decision: persist the Eventor link on the competition row

Add `competitions.eventor_event_id INTEGER NULL`. When set, every
Eventor surface uses it transparently; when null, current behaviour
(per-click picker, no badges) is preserved.

Three entry points fill the column:

1. **Wizard step 1 — Eventor quickstart.** Operator pastes the ID;
   wizard fetches `/api/eventor/events/:id`, pre-fills name + date,
   captures the ID. Wizard POST persists everything atomically.

2. **PATCH /api/competitions/:id.** Operator on an existing competition
   row uses the `[Relink]` affordance to bind / rebind / unbind. No
   cascade-delete of previously imported entries — operator may relink
   intentionally (sibling event with the same startlist, or correcting
   a wrong-event guess).

3. **Manual operator entry in the Tävling list.** A small inline edit
   on the competition card (same affordance as the relink path) for
   linking competitions that pre-date this plan.

## Why Phase 2.1, not Phase 2.0

The per-click picker already works (Plan 02-04 / ImportRunnersView).
4-klubbs MVP can ship without this. The autocomplete blocker is fixed
by Plan 02-09 (SI card duplicates); the linkage is a UX win for
sanctioned events, which 4-klubbs is not (it's a club training event,
not in Eventor's national event database).

Phase 2.1's roadmap entry ("Sanctioned-competition foundations") owns
this work. Pushing it there avoids stretching the 2.0 PR.

## Non-goals

- **Auto-push results to Eventor.** Write-side Eventor integration
  (sending FartOL's ResultList back) is a separate plan; this one
  only stores the ID.
- **Eventor classes / clubs sync per competition.** FartOL already
  pulls clubs via the cachedcompetitors boot job (Plan 02-01) and
  classes come from courseData (Plan 02-01 / 02-02). No additional
  per-event Eventor fetches needed.
- **Multi-event linkage.** One competition row → one Eventor event ID.
  Multi-stage events with shared registrations are out of scope (push
  to Phase 4 — "multi-arena, radio controls").

</objective>

<task type="auto" tdd="true">

### Task 1 — Migration 0004: `competitions.eventor_event_id`

Standard ALTER TABLE + partial index. Generator-driven via Drizzle; verify
the emitted SQL contains exactly the column + index and nothing else.

### Task 2 — `GET /api/eventor/events/:id` route + tests

Fastify handler, Zod-validated path param, calls Eventor with the
existing apiKey resolver. Maps Eventor 404 → `not_found`, 403 →
`forbidden`, key absent → `no_key`, network errors → `eventor_down`.
Return shape: `{ eventId, name, startDate, organisation }`.

### Task 3 — Wizard step 1: Eventor quickstart path

New `[Eventor]` button next to the existing courseData drop. Opens an
inline form with one input + Hämta button; calls the new route, pre-fills
name + date on success, stores `eventor_event_id` in wizard state.
Persists through to the final POST.

### Task 4 — Competition CRUD: accept + persist `eventor_event_id`

`POST /api/competitions`, `POST /api/competitions/from-wizard`, and
`PATCH /api/competitions/:id` all accept the optional column. PATCH
allows null (unlink). Tests cover round-trip + null re-write.

### Task 5 — `ImportRunnersView` collapse + relink

When the loaded competition has `eventor_event_id`, render the compact
linked-event card with `[Import startlist]` + `[Relink]` buttons.
Relink reopens the date picker; on selection PATCH the column and
re-render. Existing not-linked path stays as fallback.

### Task 6 — Tävling list: linked-event chip

`CompetitionList.svelte` renders a small Eventor chip on each linked
competition card. Inline `[Relink]` from the list opens the same picker
as ImportRunnersView (shared component).

### Task 7 — i18n keys + smoke test

Sv + en translations for the wizard quickstart, the linked-card,
the relink picker, and the list chip. Manual end-to-end: create a
new competition via the Eventor quickstart, observe the chip on the
Tävling list, click into ImportRunnersView, click `[Import startlist]`
without re-picking the event.

</tasks>

## Trust Boundaries

`/api/eventor/events/:id` becomes a new outbound surface that talks to
Eventor with the operator's API key. The key is the same one Plan 02-01
already uses; no new credential. The route returns Eventor data verbatim;
no PII concerns beyond what `cachedcompetitors` already exposes.
