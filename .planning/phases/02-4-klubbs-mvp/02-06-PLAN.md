    ## When something breaks
      Cover the failure-fallback matrix (each row a sub-section):

      ### fartOLa crashes mid-event
        1. Restart `fartola` with the same competition id.
        2. The bridge resumes from the SQLite event log (Phase 1 REQ-OPS-002).
        3. MOP receiver auto-merges MeOS-side direktanmälningar registered during the outage (D-MOP-3); a toast appears: "N löpare hämtade från MeOS".
        4. **D-LIM-1 manual workaround**: MOP `<cmp>` does NOT carry the hired flag. Hyrbrickor MARKED in MeOS during the outage will NOT auto-import. Re-enter those rentals manually in fartOLa's WalkupModal (Hyrbricka checkbox + phone) after restart.

      ### MeOS crashes mid-event
        - fartOLa keeps running standalone (Phase 1 functionality unaffected).
        - Restart MeOS; configure MIP/MOP again (config doesn't persist across MeOS restart in some installs — check the MeOS version's behavior).
        - fartOLa's MIP backlog catches MeOS up via the `lastid > 0` semantics (D-MIP-2).

      ### Eventor cache offline / API key missing
        - TweaksPanel shows red "Eventor: offline" or "nyckel saknas".
        - WalkupModal falls back to the SI-firmware `cardHolderHint` (Phase 1 behavior); operator types name+klubb manually.
        - **Never blocks the bridge** (D-EV-3, REQ-OPS-001).

      ### LAN connectivity drops between laptops
        - Both bridges keep running independently.
        - When connectivity returns, MeOS resumes /mip polling; fartOLa resumes accepting /mop POSTs.
        - No data loss — both systems' state is intact.

      ### Reader fails / wrong port
        - Check `/dev/ttyUSB0` permissions (Phase 1 Plan 18 udev rule).
        - Restart `fartola` after fixing.

      - Export final results: /competition/<id>/export (Phase 1 IOF XML 3.0 export).
