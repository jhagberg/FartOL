Land the SI5 (legacy, REQ-HW-002), SI9/SI10/SIAC modern (REQ-HW-001) decoders, register them in TWO NON-OVERLAPPING registries keyed by detection command (SI5_DET vs SI8_DET) per codex review #4, and prove every decoder against upstream's own fixture data (`{cardData, storageData}` pairs). After this plan, given a byte buffer for any of the four supported card types, the decoder produces a deeply-equal cardData object.

**KEY DESIGN CHOICES FROM CODEX REVIEW:**

- **Review #3 (HIGH):** Modern card punches live on pages 4-7 (RESEARCH §Card Decoders). The typeSpecificReadPunches chain MUST issue GET_SI8(page=4) for any card with punches; a fixture with >32 punches exercises pages 5-7 too. Acceptance grep asserts the literal `0x04` page argument in the read chain.
- **Review #4 (HIGH):** Two separate registries. SI5 is registered on `si5DetectionRegistry` (consulted only when `message.command === SI5_DET`). SI9/SI10/SIAC are registered on `si8DetectionRegistry` (consulted only when `message.command === SI8_DET`). This eliminates the cross-command capture risk. An explicit test asserts that an SI8_DET frame with cardNumber 50000 (in legacy SI5 range) does NOT instantiate SiCard5.

Purpose: This is the heart of REQ-HW-001 + REQ-HW-002 — the actual card decoding. The fixture-based test strategy from D-18 ("both upstream + ours") starts with the upstream side here: we copy upstream's `siCard5Examples`/`siCard9Examples`/`modernSiCardExamples` modules verbatim, feed their `storageData` into our decoders, and assert deepStrictEqual against their `cardData`. This buys regression coverage for free.

SIAC dispatch is heuristic (card-number range) and documented in code per the Landmine #4 mitigation. The decoders are PURE — they take bytes, return decoded objects; no I/O, no NDJSON.
