Lock the CRC behavior with the 10 frozen test vectors from RESEARCH.md — every later wave depends on these being correct.

**KEY CONTRACT CHANGE FROM UPSTREAM (codex review #1, HIGH):** Upstream uses `console.warn` for CRC failures. We replace that with a structured frame-error channel — `parseAll` accepts an optional `{onFrameError}` callback, OR returns `{messages, remainder, errors}` (the implementation chooses the callback shape since it composes better with the multiplexer's event loop). This single contract change cascades: Plans 04 and 05 no longer need to intercept `console.warn`; they wire the callback directly.

Purpose: REQ-HW-004 (CRC16-CCITT-0x8005 validation on every frame). The CRC is THE most-likely-to-silently-break component in Phase 0 (RESEARCH §Landmines #3); locking it with frozen vectors as the FIRST tests written is the single highest-value risk mitigation in this phase. Subsequent waves (card decoders, station handshake, NDJSON output) all operate on `parse`-produced messages and on the typed FrameError objects.

Output: A passing test suite for the protocol layer with 10 CRC vectors + frame parse/render coverage + structured FrameError emission + synthetic fixtures for truncation/CRC-mismatch/bad-STX.   };

  export type ParseAllOptions = {
    onFrameError?: (err: FrameError) => void;
