## Design System

| Property | Value |
|----------|-------|
| Tool | none (manual tokens — SvelteKit is not in the shadcn React ecosystem) |
| Preset | not applicable |
| Component library | none — hand-rolled Svelte components against the token contract below. Skeleton / shadcn-svelte explicitly NOT adopted in Phase 1 (CONTEXT.md "Claude's Discretion" listed them as candidates; the Claude Design prototype proved native CSS is sufficient). |
| Icon library | inline SVG only. The brand mark is a hand-drawn orienteering control flag (white-and-orange rotated diamond, locked in chat1.md). Status glyphs (`✓ ✗ ▢ ⚠ ★ ▲ ⤢`) are unicode characters by design — they print cleanly on a thermal printer. NO icon library dependency. |
| Font (default) | IBM Plex Sans (UI) + IBM Plex Mono (numbers / SI / splits). Both axes 400/500/600/700 weights. Loaded via Google Fonts subset with `display=swap`. Tabular figures + slashed zero are LOCKED requirements — chat1.md rationale: SI card numbers and split times must align in columns and `0 ≠ O`. |
| Font (operator-runtime override) | A `font-pair` runtime toggle persists to localStorage. Four pairs: `plex` (default), `geist`, `source` (Source Sans 3 + JetBrains Mono), `atkinson` (Atkinson Hyperlegible + JetBrains Mono — the bright-sun accessibility pair, REQ-UI-007). All four pairs MUST be preloaded so the toggle is instant. |
| Color space | `oklch()` for all design tokens. Locked in styles.css. NOT hex (except in `contrast-high` overrides where pure black/white is intentional). Fallback to `color-mix(in srgb, ...)` for soft halos. |

**Library policy:** Do not introduce a UI component library in Phase 1. The
prototype demonstrates the design works against ~400 lines of native CSS;
introducing Skeleton or shadcn-svelte now would couple the design contract
to an upstream's choices. Re-evaluate at Phase 2 if Yjs presence cursors
demand it.

## Spacing Scale

Declared values (multiples of 4, sourced from prototype CSS literals):

| Token | Value | Usage |
|-------|-------|-------|
| 2xs | 4px | Inline gap (status pill icon to label, brand mark to wordmark) |
| xs | 8px | Compact element spacing (nav-item gap, button icon→label gap) |
| sm | 12px | Card-head element gap, station-card row padding |
| md | 16px | Default element spacing (modal field gap, primary input padding-x) |
| lg | 18px | Card-body padding, readout column gap |
| xl | 24px | Content padding (`.content`), large hero outer padding |
| 2xl | 32px | Section-level break (hero internal padding-y) |
| 3xl | 48px | Wizard / modal generous breathing |
| 4xl | 64px | Page-level (reserved; not used in Phase 1) |

**Locked layout constants:**

| Constant | Value | Source |
|----------|-------|--------|
| `--hit` (min touch target) | **44px** | REQ-UI-007 + chat2.md responsive notes. All buttons, nav-items, inputs, selects, and history rows must satisfy `min-height: 44px`. |
| Sidebar width | 240px | `grid-template-columns: 240px 1fr` (locked) |
| Top bar height | 56px | `.topbar` fixed |
| Readout right rail | 380px (1280px+) → 340px (below 1280px) | `.readout` grid |
| Receipt paper width | 300px on screen / 80mm physical | ESC/POS thermal column |
| Border radius `--radius` | 8px | inputs, buttons, small cards |
| Border radius `--radius-lg` | 12px | cards, modals (modal uses 14px override) |

**Exceptions:** Receipt internals use sub-grid (1.5px dashed separators,
1px row padding) and are exempt from the 4px scale — the receipt is
emulating an 80mm thermal column, not the UI canvas.

**Responsive breakpoints (Phase 1 — DESKTOP FIRST, LOCKED):**

| Breakpoint | Behavior | Phase |
|------------|----------|-------|
| ≥1280px | Full readout grid (1fr / 380px) | Phase 1 (primary target) |
| 1024–1279px | Side rail shrinks to 340px | Phase 1 |
| 768–1023px (iPad portrait) | Acceptable degraded but not optimized | Phase 1 (must not break) |
| <768px | Out of scope — Phase 1.5 responsive pass | DEFERRED |

The chat2.md exchange asked whether phone/tablet work was in scope.
Decision (locked here): **Phase 1 ships desktop-first**. REQ-UI-001 says
"must work on tablet form factor"; that requirement is interpreted as
"layout must not visually break on iPad landscape (1024×768) and Chrome
Android in landscape on a 10"+ tablet." Sub-768px phone optimization is
the Phase 1.5 deliverable. The readout view is a fixed-table operator
surface — it lives on a 13"+ laptop or all-in-one PC by the finish line.

## Typography

Exactly **4 sizes** and **2 weights** (plus `font-weight: 700` reserved
for receipt headings — see "Receipt-specific typography" below):

| Role | Size | Weight | Line Height | Family | Usage |
|------|------|--------|-------------|--------|-------|
| Caption | 12px | 500 | 1.4 | UI sans | `.brand-tag`, `.station-card` rows, `.faint` mono labels, nav section header, status pill text |
| Label | 14px | 500 | 1.45 | UI sans | Button text, field labels, nav-item, history row meta, breadcrumb, table body |
| Body | 16px | 400 | 1.45 | UI sans | Default paragraph, input value, modal body, drop-zone CTA |
| Heading | 22px | 600 | 1.2 | UI sans | `.h1`, brand wordmark, runner name (28px override in readout — see below) |
| Display | 28–44px | 500–600 | 1.0 | Mix | `.h0` page title (28px, 600), runner card number (44px mono, 500), runner result time (36px mono, 500). Density "low" uses 64px/34px/52px. |

**Letter spacing:**
- Display / heading: `-0.01em` to `-0.02em` (Plex/Geist tight)
- Caption uppercase labels: `0.06em` to `0.08em`
- Mono numbers: 0 (use built-in tracking)

**Font-feature-settings:** ALL mono usages must declare
`font-feature-settings: "tnum" 1, "zero" 1;` (tabular figures + slashed zero).
Implemented as the `.mono` utility class. Locked.

**Receipt-specific typography** (deviates from main UI scale — the receipt
mimics an 80mm ESC/POS thermal column, not the screen):

| Element | Size | Weight |
|---------|------|--------|
| `.rcpt-title` | 13px | 700 |
| receipt body | 11.5px | 400 |
| receipt total | 14px | 700 |
| receipt foot | 10.5px | 400 |
| receipt mono columns | 11.5px (kids splits 9.5px) | 400 |

These are intentionally smaller than the UI minimum — the receipt is a
**visual emulation of paper output**, not part of the operator UI hit
surface. The thermal printer renders at its own DPI.

Default accent is `--accent` forest green (`oklch(0.50 0.08 145)`).
Accent is used ONLY for:

2. Active nav-item indicator (left bar + soft background)
3. Active wizard step (number badge + step card border)
4. Active history row (left bar + soft background)
5. Active receipt-template tab (text + box-shadow)
6. Drop-zone hover state border
7. Live results active class tab (bottom border + text color)

10. `<input>` focus outline (2px outline-offset -1px)
11. Toast / flash animations (`flashIn` keyframe background)

Accent is NEVER used for: body text color, generic borders, table row
backgrounds, secondary buttons, or status pills (status pills get their

### High-contrast bright-sun mode (REQ-UI-007)

Operator toggle. When `.contrast-high` is applied to the root:
- All backgrounds → pure `#ffffff` or `#f1f1f1`
- All foregrounds → pure `#000000`
- All borders → pure `#000000` (1px)
- Accent → `#00541e` (extra-dark forest)
- Status pills get a 1.5px solid border (already-locked color)
- All shadows collapse to flat 1–2px black box-shadows

This is a per-laptop / per-operator preference, persisted to localStorage.
Designed for "noon sun, glaring 13" screen at the finish line."

**Locale:** Swedish primary (`sv`), English secondary (`en`). Both
bundled in the build. Default `lang="sv"` on the HTML root. Operator can
flip via Tweaks panel; choice persists to localStorage. Every label is

`apps/web/src/lib/i18n/{sv,en}.json` and MUST be a direct port of
`01-SKETCHES/claude-design-bundle/project/i18n.js` (302 lines, ~150 keys).

Key copywriting decisions locked here so the executor doesn't drift:

**Error states:**

| Failure | Heading (sv) | Body (sv) | Recovery path |
|---------|--------------|-----------|---------------|

**Destructive actions** in Phase 1:

| Action | Confirmation copy | Confirmation approach |
|--------|-------------------|----------------------|
| Cancel walk-up registration (`Avbryt`) | (immediate, no confirm) | Clicking scrim or `Avbryt` discards the form; the read sits in history as `unmatched` and can be re-clicked. Cheap to undo. |
| Cancel wizard (`Avbryt`) | (immediate, no confirm) | No competition state persisted until step 3 completes. |
| Delete competition | NOT in Phase 1 | Out of scope. Competitions persist until explicitly purged (Phase 2 admin) or 30-day GDPR cleanup (REQ-PRIV-002). |
| Manual DNF override | `Markera som bröt?` (sv) / `Mark as DNF?` (en) | Inline confirm popover from the readout view; emits `manual_dnf` event (D-12). Reversible by emitting an `un_dnf` event. |

There are **no irreversible destructive actions** in Phase 1. The event
log is append-only (REQ-EVT-002 / D-09); every "change" is a new event,
never an UPDATE.

**Tone:** Plain Swedish, no jargon, no humor in error states. Mirrors
MeOS register where possible (`Tävlingskvitto`, `Avläsning`, `Klass`,
`Bricka`). Avoid English loanwords (e.g. `bricka` not `chip`,
`avläsning` not `readout` in Swedish surfaces).

**The "Skogis" exception:** The kids' receipt (Barn template) uses
playful Swedish — `Kläckt på`, `Spara kvittot — skogisen är din!`,
species names (`Skog`, `Stig`, `Lingon`, `Bäck`, `Skymn`, `Sol`, `Mosse`,
`Sten`). This tone is deliberate and locked.

## Interaction Contracts

### Click 1, Click 2, Click 3 — the three-click wizard (D-15 LOCKED)

The literal contract:

1. **Click 1:** `+ Ny tävling` button on home → wizard modal opens at
   step 1 (name + date fields). The "Nästa →" button advances to step 2.
2. **Click 2:** Click the drop-zone (or drag file onto it) → file picker
   opens → user picks a Purple Pen `.xml` (or IOF XML 3.0 CourseData) →
   file is validated → classes chips render below the drop-zone with a
   green "Importerad: filename.xml · Purple Pen · 6 klasser · 47 kontroller"
   confirmation. The "Nästa →" button advances to step 3.
3. **Click 3:** Step 3 auto-runs reader detection on entry. After ~1.6s
   the BSM7/8 handshake completes and `▶ Starta avläsning` becomes
   enabled. Clicking it persists the competition, closes the wizard, and
   routes to `/competition/[id]/readout`.

Each step's "Nästa →" is **disabled** until the step's prerequisite is
met. The drop-zone is empty-state by default; once filled, it switches
to `has-file` style (solid green border, OK background). Re-clicking
the drop-zone replaces the file (no confirmation needed — re-import is
cheap).

**File import flow** (locked from prototype + chat resolution):
- Accepted extensions: `.xml` only (Purple Pen produces `.xml`; IOF XML
  3.0 CourseData is also `.xml`). The format is auto-detected from XML
  root element.
- Validation happens client-side (XML parse + root element check). The
  actual schema validation happens server-side via the bridge `/api/courses/import`
  endpoint. On client-side parse failure: error state in the drop-zone
  with "Filen kunde inte läsas — förväntar Purple Pen .xml eller IOF XML
  3.0 CourseData."
- On server-side schema failure: the import is rolled back (no
  competition created) and the same error surface renders the server's
  message.

### Readout view live behavior

**WebSocket subscription** (D-13): On mount, `/competition/[id]/readout`
opens `ws://localhost:3000/ws` and subscribes to channel
`readout:<competitionId>`. Messages arrive as
`{type: 'card_read', payload: {...}, seq: N}` envelopes.

**On `card_read` message:**
1. Append payload to `history` (top of list, cap at 12 visible).
2. Set `currentRead` to the new payload.
3. Trigger `flashIn` keyframe animation on the latest-read card
   (1.6s, accent-soft → transparent).
4. If `payload.unknown === true`: open `<WalkupModal>` after 600ms delay
   (gives the flash time to register before the modal scrims it).
5. If `payload.unknown === false` AND `competition.auto_print === true`:
   trigger print path after 400ms (toast fires, ESC/POS bytes go to
   `/dev/usb/lp0`).

**On `connection_changed` message:**
- `connected: true` → pulse-dot green, label "Ansluten · puls Xms"
- `connected: false` → pulse-dot red, label "Frånkopplad", auto-reconnect
  in background

**Click any history row:** that read becomes `currentRead` (re-renders
the latest-read card and the receipt mirror below). The clicked row gets
the `.active` accent-soft background + left bar.

**Auto-reconnect:** Small client wrapper around native WebSocket
(per D-13 Claude's Discretion). On disconnect: 1s/2s/4s/8s backoff up to
30s. On reconnect: emit a `hello` message identifying last-seen `seq`;
server replays missed messages from the SQLite event log (REQ-EVT-002
guarantees this is possible).

### Live results auto-update (LOCKED)

The chat noted the prototype only updates on manual simulate. The Phase
1 contract:

- `/competition/[id]/results` subscribes to `results:<competitionId>`.
- Every `card_read` (any class) re-derives the result projection
  server-side (REQ-EVT-EVT-003 reducer) and pushes a
  `{type: 'results_update', payload: {classId, rows: [...]}}` message.
- Client receives → re-renders the affected tab's table.
- The just-added row gets the `.new` accent-soft background for 4
  seconds, then fades to neutral.
- Sort stability: rows re-sorted on each update; pending starters always
  sort last; OK starters sort by elapsed time ascending.
- "Updated 14:32:11" timestamp in the header updates on every message.

**Fullscreen mode** (`<F11>`-equivalent button): scales typography up
(plc 22px, name 18px, time 22px), no chrome change. Designed for
projection at 1080p. Operator-toggled; not auto-entered.

### Walk-up modal (REQ-EVT-CMP-004, D-04)

Triggered automatically when an `unknown: true` card is read. **Blocks
only the current card** (the rest of the readout view stays interactive
under the scrim). Subsequent unknown cards queue (newest replaces
current; the previous unknown card stays in history as `unmatched` and
can be clicked later to re-open the walk-up form).

Fields:
- **Namn** (required) — placeholder "För- och efternamn", minLength 2,
  trimmed. No regex (Swedish names span å/ä/ö/-/' freely).
- **Klubb** (optional) — placeholder "Frivilligt — t.ex. StorTuna OK".
  Autocomplete from past entries (`window.CLUBS` in prototype; SQLite
  `competitor_clubs` aggregate in production). Free-text allowed.
- **Klass** (required) — `<select>` of classes imported for this
  competition. No default selection (forces operator choice).
- **Bricka** (pre-filled from read, editable) — operator can correct a
  misread by hand-typing the SI number. Integer only, minimum 1.

On `Spara och bind`:
1. POST `/api/competitors` with `{name, club, classId, cardNumber, competitionId}`.
2. Server inserts competitor row + emits `card_bound` event over the
   readout channel.
3. Client closes modal; readout view re-renders with the now-bound
   competitor as `currentRead`.

On `Avbryt`: modal closes, the unknown card stays in history as
`unmatched` and can be retried.

### Export IOF XML 3.0 ResultList (REQ-EVT-CMP-008, REQ-STD-002)

The prototype showed an `Export` nav-item with a `IOF 3.0` badge but no
screen. Phase 1 contract LOCKED here:

- Route: `/competition/[id]/export`
- Surface: single page, NOT a modal. Sections:
  1. **Export type** — radio: `Slutgiltig` (final, default) / `Provisorisk`
     (provisional). Provisional is allowed mid-event for sharing.
  2. **Validation status** — green/red box. On entry, the page calls
     `GET /api/competitions/[id]/export/preview` which runs the XSD
     validation server-side and returns either `{valid: true, summary: {...}}`
     or `{valid: false, errors: [{line, message}]}`. Errors render
     inline with line numbers.
  3. **Download** — primary CTA: `↓ Hämta ResultList.xml` (sv) / `↓ Download ResultList.xml` (en).
     Enabled ONLY if validation passed. Clicking calls
     `GET /api/competitions/[id]/export?format=iof30` which streams the
     XML file with `Content-Disposition: attachment; filename="{slug}-resultlist.xml"`.

Validation MUST happen before the file is written to disk (SC#6 binding
contract: "validate against XSD before saving"). Failed XSD validation
is non-destructive: no file written, error surfaced inline, operator can
re-trigger after fixing data.
