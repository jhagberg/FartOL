Wizard state shape (Svelte 5 runes inside NewCompetitionWizard.svelte):
```
let state = $state({
  step: 1 as 1 | 2 | 3,
  name: '',
  date: '',                   // YYYY-MM-DD text
  fileMetadata: null as { filename: string; kind: 'CourseData' | 'EntryList'; class_count: number; control_count: number } | null,
  competitionId: null as string | null,   // created at end of step 1, BEFORE step 2's file import (so import POSTs to /api/competitions/:id/import)
  importError: null as string | null,
  readerStatus: 'opening' as 'opening' | 'open' | 'error',
  readerHandshakeError: null as string | null,
});
```

Order of operations: per UI-SPEC §Wizard, the flow is "Create → Import course → Ready". The competition is created EAGERLY on step 1 → step 2 transition so the import endpoint has a competition_id. If the user clicks Avbryt at step 2 or 3, the partially-created competition is DELETED via... wait. UI-SPEC §"Destructive actions": NO delete competition in Phase 1. RESOLUTION: defer the createCompetition POST until the final Starta-avläsning click. Step 2 client-side validates the XML root element (no server POST) and stores the file bytes in memory. On step 3 click, the wizard POSTs createCompetition + importFile + setActiveCompetition sequentially. If anything fails, the wizard surfaces the error inline; no orphan competition.

ALTERNATIVE: create competition eagerly. If user clicks Avbryt, the empty competition lingers (no DELETE). UI-SPEC §"Destructive actions" says explicitly the wizard Avbryt "no competition state persisted until step 3 completes." → THE LOCKED CONTRACT IS DEFERRED POST. Implement it this way.

Reader-detect step (step 3): Per UI-SPEC §"Click 1, Click 2, Click 3" — "Step 3 auto-runs reader detection on entry. After ~1.6s the BSM7/8 handshake completes and `▶ Starta avläsning` becomes enabled."

In dev (without hardware), we simulate this:
- On entry to step 3, the wizard sets readerStatus='opening'.
- After 1600ms, IF in dev mode (`import.meta.env.DEV` or `?dev=1` query param), force readerStatus='open' regardless of real reader status. This lets Playwright tests pass without /dev/ttyUSB0.
- In production: the wizard subscribes to a global ws envelope on channel `bridge:status` (a new convention) OR queries GET /api/sessions/bridge-status (a new endpoint — out of scope this plan; we use a simple poll on `/api/health` extended with `bridge_state`).
- SIMPLIFICATION for Phase 1: poll `/api/sessions/active-competition` every 500ms; the bridge is implicitly "open" if the bin started without --no-bridge. For the wizard, we just use the dev-timeout path (1600ms then "open"). Real BSM7/8 detection lands in plan 13 readout view via existing connection_changed events. Document this as Phase 1's accepted simplification — "the wizard pretends the reader is ready after 1.6s; the readout view is where reader status is authoritative."

Wizard navigation:
- /competition/[id]/+page.svelte is the per-competition shell. It checks for a `?wizard=1` query param: if present, NewCompetitionWizard renders as an overlay. Otherwise it's a competition-detail surface.
- Home's "+ Ny tävling" navigates to `/competition/_new?wizard=1` (literal id "_new" sentinel) which mounts the wizard with state.competitionId=null. After successful step 3, navigates to `/competition/<real-id>/readout`.

The DropZone component (a new UI primitive — adding it to apps/web/src/lib/components/ rather than ui/ since it's higher-level):
- File-drop event handlers (dragover, drop, click) + a hidden <input type="file" accept=".xml,application/xml">.
- States: idle / dragover / has-file / error.
- Emits onFileSelected(file: File) callback.
- UI-SPEC §"Visual Anchors" — re-clicking replaces the file (no confirmation).
