**Order of operations (LOCKED — deferred POST + C-H3 single atomic call):**

UI-SPEC §Wizard + §"Destructive actions" lock the deferred-POST contract: "no competition state persisted until step 3 completes." Step 2 client-side validates the XML root element (no server POST) and stores the file bytes in `state.pendingFile`.

On step 3's `▶ Starta avläsning` click, the wizard makes ONE atomic POST:

1. Read state.pendingFile as bytes; convert to base64 (browser `FileReader.readAsDataURL` or `Uint8Array.toBase64()` polyfill — executor's choice).
2. POST `/api/competitions/from-wizard` body `{ name, date, xml_file: { name: state.pendingFile.name, content_base64 } }`. The endpoint (plan 05 owns) wraps competition INSERT + XML ingest in a single SQL transaction.
3. On 201 → state.competitionId = response.competition_id.
4. POST setActiveCompetition(state.competitionId).
5. goto(`/competition/${state.competitionId}/readout`).

If step 2 (the from-wizard POST) returns 4xx/5xx, the wizard surfaces the inline error and stays open. **No orphan competition rows ever exist** — the server's transaction rolled back. There is no DELETE-competition gesture in Phase 1; none is needed because the atomic endpoint mathematically prevents orphan rows.

Avbryt at any step closes the modal cleanly — non-destructive (no POSTs fired yet OR atomic transaction rolled back).

Reader-detect step (step 3): unchanged from prior revision. Dev mode forces readerStatus='open' after 1600ms; real bridge state lands authoritatively in plan 13 readout view.
